# JavaInfoHunter 数据库设计说明

## 概述

JavaInfoHunter 使用 PostgreSQL 16+ 作为主数据库，配合 pgvector 扩展实现向量相似度搜索。数据库设计遵循第三范式，优化查询性能，并支持 AI 原生特性。

## 技术栈

- **数据库**: PostgreSQL 16+
- **向量扩展**: pgvector (用于语义搜索)
- **迁移工具**: Flyway
- **ORM**: Spring Data JPA (Hibernate)
- **连接池**: HikariCP

## 表结构详解

### 1. rss_sources (RSS 订阅源表)

**用途**: 存储 RSS 订阅源信息及爬取配置

**主键**: `id` (BIGSERIAL)

**核心字段**:
- `url` (VARCHAR 2048, UNIQUE): RSS feed URL，唯一约束防止重复
- `name` (VARCHAR 255): 人类可读的源名称
- `category` (VARCHAR 100): 分类（如 technology, business）
- `tags[]` (TEXT[]): 标签数组，灵活分类
- `is_active` (BOOLEAN): 是否活跃爬取
- `crawl_interval_seconds` (INTEGER): 爬取间隔（秒）
- `last_crawled_at` (TIMESTAMP): 最后爬取时间

**统计字段**:
- `total_articles` (BIGINT): 总文章数
- `failed_crawls` (BIGINT): 失败次数

**索引**:
- `idx_rss_sources_active`: 活跃源查询
- `idx_rss_sources_category`: 分类查询
- `idx_rss_sources_tags`: GIN 索引，标签搜索
- `idx_rss_sources_last_crawled`: 爬取调度

**约束**:
```sql
CONSTRAINT chk_crawl_interval_positive CHECK (crawl_interval_seconds > 0)
CONSTRAINT chk_max_retries_positive CHECK (max_retries >= 0)
```

### 2. raw_content (原始内容表)

**用途**: 存储从 RSS 抓取的原始内容，AI 处理前

**主键**: `id` (BIGSERIAL)

**外键**:
- `rss_source_id` (FK → rss_sources.id): 关联 RSS 源

**核心字段**:
- `guid` (VARCHAR 255): RSS feed 中的唯一标识
- `title` (VARCHAR 500): 文章标题
- `link` (VARCHAR 2048): 文章链接
- `raw_content` (TEXT): 原始 HTML 或纯文本
- `content_hash` (VARCHAR 64, UNIQUE): SHA-256 哈希，全局去重
- `processing_status` (VARCHAR 20): 处理状态 (pending/processing/completed/failed)

**向量字段**:
- `embedding` (vector(1536)): OpenAI embedding 向量
- `embedding_generated_at` (TIMESTAMP): 向量生成时间

**时间字段**:
- `publish_date` (TIMESTAMP): 原始发布时间
- `crawl_date` (TIMESTAMP): 爬取时间

**唯一约束**:
```sql
CONSTRAINT uq_raw_content_guid_source UNIQUE (guid, rss_source_id)
CONSTRAINT uq_raw_content_hash UNIQUE (content_hash)
```

**索引**:
- `idx_raw_content_rss_source`: 按源查询
- `idx_raw_content_status`: 处理状态查询
- `idx_raw_content_publish_date`: 按发布时间排序
- `idx_raw_content_embedding`: IVFFlat 向量索引（余弦相似度）

```sql
CREATE INDEX idx_raw_content_embedding
ON raw_content
USING ivfflat(embedding vector_cosine_ops)
WITH (lists = 100);
```

### 3. news (处理后新闻表)

**用途**: 存储 AI 处理和丰富后的新闻文章

**主键**: `id` (BIGSERIAL)

**外键**:
- `raw_content_id` (FK → raw_content.id, UNIQUE): 一对一关联

**AI 生成字段**:
- `summary` (TEXT): AI 生成的摘要
- `topics[]` (TEXT[]): AI 提取的主题
- `keywords[]` (TEXT[]): AI 提取的关键词
- `sentiment` (VARCHAR 20): 情感分类 (positive/neutral/negative)
- `sentiment_score` (DECIMAL 3,2): 情感得分 (-1.0 到 1.0)
- `importance_score` (DECIMAL 3,2): 重要性得分 (0.0 到 1.0)

**分类字段**:
- `category` (VARCHAR 100): AI 分配的分类
- `tags[]` (TEXT[]): AI 分配的标签

**SEO 字段**:
- `slug` (VARCHAR 255, UNIQUE): URL 友好的 slug
- `featured_image_url` (VARCHAR 2048): 特色图片

**互动指标**:
- `view_count` (BIGINT): 浏览次数
- `like_count` (INTEGER): 点赞数
- `share_count` (INTEGER): 分享数

**发布状态**:
- `is_published` (BOOLEAN): 是否已发布
- `is_featured` (BOOLEAN): 是否精选
- `published_at` (TIMESTAMP): 发布时间

**索引**:
- `idx_news_published`: 发布状态和时间
- `idx_news_category`: 分类查询
- `idx_news_tags`: GIN 索引，标签搜索
- `idx_news_topics`: GIN 索引，主题搜索
- `idx_news_keywords`: GIN 索引，关键词搜索
- `idx_news_sentiment`: 情感分析
- `idx_news_importance`: 重要性排序
- `idx_news_fulltext`: 全文搜索（GIN to_tsvector）

**全文搜索索引**:
```sql
CREATE INDEX idx_news_fulltext
ON news
USING GIN(to_tsvector('english', title || ' ' || COALESCE(summary, '') || ' ' || COALESCE(full_content, '')));
```

### 4. agent_executions (Agent 执行记录表)

**用途**: 追踪 AI Agent 执行，用于调试、分析和优化

**主键**: `id` (BIGSERIAL)

**核心字段**:
- `agent_id` (VARCHAR 100): 注册的 Agent ID（如 'crawler-agent'）
- `agent_type` (VARCHAR 50): Agent 类型（BaseAgent/ReActAgent/ToolCallAgent）
- `execution_id` (VARCHAR 100, UNIQUE): 唯一执行标识符（UUID）
- `status` (VARCHAR 20): 执行状态 (running/completed/failed/cancelled)

**协作模式**:
- `coordination_pattern` (VARCHAR 50): 协作模式 (chain/parallel/master-worker/standalone)
- `correlation_id` (VARCHAR 100): 关联 ID，追踪协作执行
- `parent_execution_id` (VARCHAR 100): 父执行 ID，链式追踪

**输入输出**:
- `input_data` (JSONB): 输入参数和上下文
- `output_data` (JSONB): Agent 输出和结果
- `error_trace` (TEXT): 错误堆栈追踪

**性能指标**:
- `start_time` (TIMESTAMP): 开始时间
- `end_time` (TIMESTAMP): 结束时间
- `duration_milliseconds` (INTEGER): 执行时长
- `total_steps` (INTEGER): 总步数
- `tokens_used` (INTEGER): 使用的 token 数
- `estimated_cost_usd` (DECIMAL 10,4): 估算成本（美元）

**工具使用**:
- `tools_used[]` (TEXT[]): 使用的工具列表
- `tool_call_count` (INTEGER): 工具调用次数

**关联**:
- `raw_content_id` (FK → raw_content.id): 可选，关联原始内容
- `news_id` (FK → news.id): 可选，关联新闻

**重试机制**:
- `retry_count` (INTEGER): 当前重试次数
- `max_retries` (INTEGER): 最大重试次数

**索引**:
- `idx_agent_executions_agent_id`: 按 Agent 查询
- `idx_agent_executions_agent_type`: 按类型查询
- `idx_agent_executions_execution_id`: 执行 ID 查询
- `idx_agent_executions_status`: 状态查询
- `idx_agent_executions_start_time`: 时间排序
- `idx_agent_executions_correlation_id`: 协作追踪
- `idx_agent_executions_agent_time`: 复合索引（Agent + 时间）

**复合索引示例**:
```sql
CREATE INDEX idx_agent_executions_agent_time
ON agent_executions(agent_id, start_time DESC);

CREATE INDEX idx_agent_executions_status_time
ON agent_executions(status, start_time DESC);
```

## 索引设计策略

### 1. B-Tree 索引（默认）
**适用场景**:
- 等值查询 (=, IN)
- 范围查询 (<, >, BETWEEN)
- 排序 (ORDER BY)

**示例**:
```sql
CREATE INDEX idx_news_published
ON news(is_published, published_at DESC);
```

### 2. GIN 索引（数组和全文搜索）
**适用场景**:
- 数组包含查询 (@>, ANY, &&)
- 全文搜索 (to_tsvector)

**示例**:
```sql
-- 数组索引
CREATE INDEX idx_rss_sources_tags
ON rss_sources USING GIN(tags);

-- 全文搜索索引
CREATE INDEX idx_news_fulltext
ON news USING GIN(to_tsvector('english', title || ' ' || summary));
```

### 3. IVFFlat 向量索引（pgvector）
**适用场景**:
- 向量相似度搜索
- 近似最近邻（ANN）查询

**参数说明**:
- `lists = 100`: 聚类数量（越大越精确，但内存占用越高）
- `vector_cosine_ops`: 余弦相似度距离函数

**示例**:
```sql
CREATE INDEX idx_raw_content_embedding
ON raw_content
USING ivfflat(embedding vector_cosine_ops)
WITH (lists = 100);

-- 查询示例
SELECT * FROM raw_content
ORDER BY embedding <=> '[0.1, 0.2, ...]'  -- 余弦距离
LIMIT 10;
```

**向量索引最佳实践**:
- 表大小 < 100万行：lists = 100
- 表大小 100万-1000万行：lists = 1000
- 表大小 > 1000万行：lists = 10000

### 4. 部分索引（条件索引）
**适用场景**:
- 只索引满足条件的行
- 减少索引大小和维护开销

**示例**:
```sql
-- 只索引活跃的 RSS 源
CREATE INDEX idx_rss_sources_active
ON rss_sources(is_active)
WHERE is_active = TRUE;
```

## 查询优化建议

### 1. RSS 源查询优化

**查找待爬取的源**:
```sql
-- 使用复合索引
SELECT * FROM rss_sources
WHERE is_active = TRUE
  AND (last_crawled_at IS NULL
       OR last_crawled_at < NOW() - INTERVAL '1 hour')
ORDER BY last_crawled_at ASC NULLS LAST;
```

**索引**: `idx_rss_sources_active` + `idx_rss_sources_last_crawled`

### 2. 内容去重查询优化

**检查内容是否已存在**:
```sql
-- 使用 content_hash 唯一索引
SELECT id FROM raw_content
WHERE content_hash = 'sha256_hash_here';
```

**索引**: `uq_raw_content_hash` (UNIQUE)

### 3. 向量相似度搜索优化

**查找相似内容**:
```sql
-- 使用 IVFFlat 索引
SELECT id, title, 1 - (embedding <=> '[0.1, 0.2, ...]') AS similarity
FROM raw_content
WHERE embedding IS NOT NULL
ORDER BY embedding <=> '[0.1, 0.2, ...]'
LIMIT 10;
```

**索引**: `idx_raw_content_embedding` (IVFFlat)

**性能提示**:
- 向量索引在数据量 < 1000 行时不生效（使用顺序扫描）
- 建议在表中有足够数据后再创建向量索引

### 4. 全文搜索优化

**新闻全文搜索**:
```sql
-- 使用 GIN 全文索引
SELECT id, title, summary
FROM news
WHERE to_tsvector('english', title || ' ' || summary || ' ' || COALESCE(full_content, ''))
      @@ to_tsquery('english', 'machine & learning')
ORDER BY ts_rank(to_tsvector('english', title || ' ' || summary), to_tsquery('english', 'machine & learning')) DESC
LIMIT 20;
```

**索引**: `idx_news_fulltext` (GIN)

### 5. Agent 执行统计查询优化

**Agent 性能统计**:
```sql
-- 使用复合索引
SELECT
    agent_id,
    COUNT(*) AS total_executions,
    AVG(duration_milliseconds) AS avg_duration,
    SUM(tokens_used) AS total_tokens,
    SUM(estimated_cost_usd) AS total_cost
FROM agent_executions
WHERE start_time > NOW() - INTERVAL '7 days'
GROUP BY agent_id
ORDER BY total_executions DESC;
```

**索引**: `idx_agent_executions_agent_time`

### 6. 分页查询优化

**使用游标分页（避免 OFFSET）**:
```sql
-- 传统分页（性能差）
SELECT * FROM news
WHERE is_published = TRUE
ORDER BY published_at DESC
LIMIT 20 OFFSET 1000;

-- 游标分页（性能好）
SELECT * FROM news
WHERE is_published = TRUE
  AND published_at < :last_published_at
ORDER BY published_at DESC
LIMIT 20;
```

## 数据库视图

### 1. active_rss_sources
**用途**: 活跃 RSS 源及统计信息

```sql
CREATE OR REPLACE VIEW active_rss_sources AS
SELECT
    id, name, url, category, tags,
    crawl_interval_seconds, last_crawled_at,
    total_articles, failed_crawls,
    ROUND(100.0 * failed_crawls / NULLIF(total_articles, 0), 2) AS failure_rate,
    created_at
FROM rss_sources
WHERE is_active = TRUE
ORDER BY last_crawled_at ASC NULLS LAST;
```

### 2. pending_content
**用途**: 待处理内容列表

```sql
CREATE OR REPLACE VIEW pending_content AS
SELECT
    rc.id, rc.rss_source_id, rs.name AS source_name,
    rc.title, rc.link, rc.publish_date, rc.crawl_date,
    rc.processing_status
FROM raw_content rc
JOIN rss_sources rs ON rc.rss_source_id = rs.id
WHERE rc.processing_status = 'pending'
ORDER BY rc.crawl_date ASC;
```

### 3. published_news
**用途**: 已发布新闻及完整元数据

```sql
CREATE OR REPLACE VIEW published_news AS
SELECT
    n.id, n.title, n.summary, n.category, n.tags,
    n.topics, n.keywords, n.sentiment, n.importance_score,
    n.view_count, n.like_count, n.published_at,
    rs.name AS source_name, rc.link AS original_link,
    rc.author AS original_author, rc.publish_date AS original_publish_date
FROM news n
JOIN raw_content rc ON n.raw_content_id = rc.id
JOIN rss_sources rs ON rc.rss_source_id = rs.id
WHERE n.is_published = TRUE
ORDER BY n.published_at DESC;
```

### 4. agent_execution_stats
**用途**: Agent 执行统计（最近 7 天）

```sql
CREATE OR REPLACE VIEW agent_execution_stats AS
SELECT
    agent_id, agent_type,
    COUNT(*) AS total_executions,
    COUNT(*) FILTER (WHERE status = 'completed') AS completed_count,
    COUNT(*) FILTER (WHERE status = 'failed') AS failed_count,
    COUNT(*) FILTER (WHERE status = 'running') AS running_count,
    ROUND(AVG(duration_milliseconds) / 1000.0, 2) AS avg_duration_seconds,
    ROUND(SUM(tokens_used), 0) AS total_tokens,
    ROUND(SUM(estimated_cost_usd), 4) AS total_cost_usd,
    ROUND(100.0 * COUNT(*) FILTER (WHERE status = 'failed') / NULLIF(COUNT(*), 0), 2) AS failure_rate
FROM agent_executions
WHERE start_time >= CURRENT_DATE - INTERVAL '7 days'
GROUP BY agent_id, agent_type
ORDER BY total_executions DESC;
```

## 触发器

### 自动更新 updated_at 字段

**触发器函数**:
```sql
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = CURRENT_TIMESTAMP;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

**应用到所有表**:
```sql
CREATE TRIGGER update_rss_sources_updated_at
    BEFORE UPDATE ON rss_sources
    FOR EACH ROW
    EXECUTE FUNCTION update_updated_at_column();

-- 同样应用于 raw_content, news, agent_executions
```

## 数据完整性约束

### 1. 主键约束 (PRIMARY KEY)
- 所有表都有自增主键 `id`
- 使用 BIGSERIAL 支持大数据量

### 2. 外键约束 (FOREIGN KEY)
- `raw_content.rss_source_id` → `rss_sources.id` (ON DELETE CASCADE)
- `news.raw_content_id` → `raw_content.id` (ON DELETE CASCADE)
- `agent_executions.raw_content_id` → `raw_content.id` (ON DELETE SET NULL)
- `agent_executions.news_id` → `news.id` (ON DELETE SET NULL)

### 3. 唯一约束 (UNIQUE)
- `rss_sources.url`: 防止重复的 RSS 源
- `raw_content.content_hash`: 全局内容去重
- `raw_content.guid + rss_source_id`: 同一源内去重
- `news.slug`: URL slug 唯一性
- `agent_executions.execution_id`: 执行 ID 唯一性

### 4. 检查约束 (CHECK)
- `crawl_interval_seconds > 0`: 爬取间隔必须为正数
- `sentiment_score BETWEEN -1.0 AND 1.0`: 情感得分范围
- `importance_score BETWEEN 0.0 AND 1.0`: 重要性得分范围
- `processing_status IN (...)`: 枚举值检查

## 附录：SQL 查询示例

### 1. 查找待处理的优质内容
```sql
SELECT rc.id, rc.title, rs.name AS source, rc.publish_date
FROM raw_content rc
JOIN rss_sources rs ON rc.rss_source_id = rs.id
WHERE rc.processing_status = 'pending'
  AND rs.is_active = TRUE
ORDER BY rc.publish_date DESC
LIMIT 100;
```

### 2. 查找相似新闻（向量搜索）
```sql
SELECT
    n.id,
    n.title,
    n.summary,
    1 - (rc.embedding <=> '[0.1, 0.2, ...]') AS similarity
FROM news n
JOIN raw_content rc ON n.raw_content_id = rc.id
WHERE n.is_published = TRUE
  AND rc.embedding IS NOT NULL
ORDER BY rc.embedding <=> '[0.1, 0.2, ...]'
LIMIT 10;
```

### 3. Agent 性能报告
```sql
SELECT
    agent_id,
    COUNT(*) AS executions,
    ROUND(AVG(duration_milliseconds / 1000.0), 2) AS avg_duration_sec,
    ROUND(SUM(tokens_used), 0) AS total_tokens,
    ROUND(SUM(estimated_cost_usd), 2) AS total_cost,
    ROUND(100.0 * COUNT(*) FILTER (WHERE status = 'failed') / COUNT(*), 2) AS failure_rate
FROM agent_executions
WHERE start_time > NOW() - INTERVAL '30 days'
GROUP BY agent_id
ORDER BY total_cost DESC;
```

### 4. 热门话题趋势
```sql
SELECT
    unnest(topics) AS topic,
    COUNT(*) AS article_count,
    AVG(importance_score) AS avg_importance
FROM news
WHERE published_at > NOW() - INTERVAL '7 days'
  AND is_published = TRUE
GROUP BY topic
ORDER BY article_count DESC
LIMIT 20;
```
